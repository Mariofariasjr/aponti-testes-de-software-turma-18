# Atividade Avaliativa – Comparativo entre Metodologias Ágil e Cascata

## Objetivo

Comparar as metodologias **Ágil** e **Cascata**, destacando como os testes de software são realizados em cada modelo, além de apresentar suas principais vantagens e limitações.

---

# Tabela Comparativa

| Aspecto | Ágil | Cascata |
|---------|------|----------|
| **Modelo de desenvolvimento** | Iterativo e incremental | Sequencial e linear |
| **Planejamento** | Flexível e adaptável | Definido no início do projeto |
| **Entrega** | Pequenas entregas frequentes (Sprints) | Entrega única ao final do projeto |
| **Mudanças nos requisitos** | São aceitas e adaptadas durante o desenvolvimento | São difíceis e podem gerar alto custo |
| **Participação do cliente** | Contínua durante todo o projeto | Geralmente ocorre no início e na entrega final |
| **Documentação** | Essencial, porém mais enxuta | Completa e detalhada |
| **Feedback** | Constante | Normalmente apenas ao final do projeto |

---

# Como os testes acontecem em cada modelo

## Ágil

Os testes acontecem durante todo o desenvolvimento. A cada Sprint, novas funcionalidades são desenvolvidas e testadas, permitindo identificar e corrigir problemas rapidamente.

### Características

- Testes contínuos
- Automação de testes quando possível
- Integração entre QA e equipe de desenvolvimento
- Feedback rápido
- Correção de defeitos ainda durante a Sprint

---

## Cascata

Os testes são realizados somente após a conclusão da fase de desenvolvimento. Caso sejam encontrados defeitos, as correções podem gerar retrabalho e aumentar o tempo do projeto.

### Características

- Testes concentrados na fase final
- Maior tempo para encontrar defeitos
- Correções podem ser mais caras
- Processo bem definido

---

# Vantagens e Limitações

| Metodologia | Vantagens | Limitações |
|--------------|-----------|------------|
| **Ágil** | Entregas rápidas, feedback constante, maior adaptação às mudanças, maior colaboração da equipe | Requer comunicação constante, planejamento contínuo e participação ativa do cliente |
| **Cascata** | Organização, documentação detalhada, fácil acompanhamento das etapas e previsibilidade | Pouca flexibilidade para mudanças, testes tardios e maior risco de retrabalho |

---

# Comparação dos Testes

| Critério | Ágil | Cascata |
|----------|-------|----------|
| Quando os testes são realizados? | Durante todo o desenvolvimento | Após finalizar o desenvolvimento |
| Frequência dos testes | Contínua | Apenas na fase de testes |
| Correção de defeitos | Imediata | Após a conclusão do desenvolvimento |
| Feedback | Constante | No final do projeto |
| Automação de testes | Muito utilizada | Menos comum |

---

# Conclusão

As metodologias Ágil e Cascata possuem características diferentes. No modelo Ágil, os testes fazem parte de todas as etapas do desenvolvimento, permitindo encontrar e corrigir problemas rapidamente. Já no modelo Cascata, os testes são realizados apenas após a conclusão do desenvolvimento, o que pode aumentar o custo e o tempo para corrigir defeitos.

Para projetos que exigem rapidez, adaptação e entregas frequentes, o modelo Ágil costuma ser a melhor escolha. Já o modelo Cascata é mais indicado para projetos com requisitos bem definidos e poucas mudanças durante o desenvolvimento.

---

# Reflexão

Como estudante de QA, percebi que a qualidade do software depende da forma como os testes são executados. No modelo Ágil, o QA participa desde o início do projeto, ajudando a prevenir defeitos. No modelo Cascata, os testes são importantes, porém acontecem apenas no final, o que pode dificultar a correção de problemas. Entendi que o papel do profissional de QA é garantir que o software seja entregue com qualidade, independentemente da metodologia utilizada.
